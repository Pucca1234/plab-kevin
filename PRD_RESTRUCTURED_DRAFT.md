## 1. 문서 목적
- 이 문서는 `Kevin Dashboard`가 무엇을 제공해야 하고, 어떤 규칙으로 동작해야 하며, 개발자가 어떤 기준으로 구현/수정/검증해야 하는지를 정의한다.
- 이 문서는 요구사항 문서이면서 동시에 기능 명세와 운영 정책의 기준 문서다.
- 구현 가이드, 실행 방법, 파일 지도, 작업 시작용 온보딩 정보는 `README_RESTRUCTURED_DRAFT.md`에서 다룬다.
- 날짜별 변경 이력은 `CHANGELOG_DRAFT.md`에서 관리한다.

## 2. 이 문서의 독자
- 신규 개발자
- 기존 개발자

- 어떤 값이 source of truth인지

## 3. 작성 규칙
### 3.1 문서 역할
- 이 문서는 “현재 이렇게 구현돼 있다”보다 “이렇게 동작해야 한다”를 우선한다.
- 단, 운영 중인 실제 구현 제약이 중요하면 요구사항 아래에 명시한다.

### 3.2 서술 원칙
- 기능은 반드시 `목적`, `입력`, `처리 규칙`, `출력/결과`, `예외/주의사항` 순으로 설명한다.
- UI 설명만 하지 말고, 실제 비즈니스 로직과 상태 전이 기준을 함께 적는다.
- “좋아 보인다”, “직관적이다” 같은 모호한 표현보다 판정 가능한 규칙으로 쓴다.

### 3.3 업데이트 원칙

- README와 중복 서술하지 않는다.

- 특정 커밋 단위의 세부 변경을 본문에 계속 쌓지 않는다.
- 실행 명령, 환경변수, 파일 목록을 과도하게 중복하지 않는다.

## 5. 시스템 경계와 source of truth
### 5.1 데이터 역할 분리
- 분석 source of truth: `BigQuery`
- 앱 운영 데이터 저장소: `Supabase`

### 5.2 BigQuery 운영 원칙
- source dataset은 read-only로 취급한다.
- 신규/재생성 객체는 `kevin_serving` dataset에만 생성한다.
- source 반영 지연이 있는 날은 serving rebuild가 당일 최신치를 모두 포함하지 못할 수 있다.

### 5.3 Supabase 역할
- 로그인/인증
- 템플릿 저장
- 사용자별 설정
- 앱 운영에 필요한 부가 데이터

## 6. 데이터 구조
### 6.1 Source tables
- `plabfootball-51bf5.data_mart.data_mart_1_social_match`
- `plabfootball-51bf5.googlesheets.metric_store_native`

### 6.2 Serving layer
- `plabfootball-51bf5.kevin_serving.weeks_view`
- `plabfootball-51bf5.kevin_serving.entity_hierarchy`
- `plabfootball-51bf5.kevin_serving.weekly_agg`
- `plabfootball-51bf5.kevin_serving.weekly_expanded_agg`

### 6.3 집계 규칙
- `cnt` 계열: `MAX(value)`
- `rate` 계열: `AVG(value)`
- 집계 그레인: `(week, measure_unit, filter_value, metric_id)`

### 6.4 all 단위 해석
- `all`은 전체 요약 레벨이다.
- 운영 검증 시 대표 지표 기준으로 하위 단위 합계나 비교 가능한 기준값과 정합성을 확인한다.

## 8. 핵심 사용자 시나리오
### 8.1 기본 조회
1. 사용자가 지표를 선택한다.
2. 기간단위와 기간범위를 선택한다.
3. 측정단위를 선택한다.
4. 기간 필터와 엔티티 필터를 좁힌다.
5. 조회한다.
6. 결과 테이블과 추이/히트맵/증감을 확인한다.

### 8.2 엔티티 원인 분석
1. 사용자가 특정 측정단위 결과를 본다.
2. 특정 엔티티를 클릭한다.
3. 하위 드릴다운 후보를 선택한다.
4. 같은 조건을 유지한 채 더 세부 단위 결과를 본다.

### 8.3 기간 원인 분석
1. 사용자가 기간 컬럼에서 특정 기간을 확인한다.
2. 해당 컬럼을 클릭한다.
3. 더 세부 기간단위를 선택한다.
4. 선택한 기간 범위 내부만 대상으로 더 세부 결과를 본다.

### 8.4 템플릿 재사용
1. 사용자가 자주 쓰는 검색 조건을 저장한다.
2. 이후 동일 조건을 빠르게 복원한다.

## 9. 기능 명세
## 9.1 지표 선택
### 목적
- 사용자가 무엇을 볼지 결정한다.

### 입력
- 지표 검색어
- 지표 체크 상태

### 처리 규칙
- 사용자는 1개 이상 지표를 선택할 수 있어야 한다.
- 지표 선택 패널은 검색 기능을 제공해야 한다.
- 지표별 설명을 확인할 수 있어야 한다.
- 선택된 지표가 없으면 조회 완료 상태를 확정할 수 없어야 한다.

### 출력/결과
- 선택된 지표 ID 집합
- 결과 테이블의 지표 행/열 구성 기준

### 예외/주의사항
- 지표 설명은 편의 기능이지만, 지표 ID와 실제 집계 의미를 혼동시키면 안 된다.

## 9.2 기간단위와 기간범위
### 목적
- 분석의 시간 축을 정의한다.

### 입력
- 기간단위: `연/분기/월/주/일`
- 기간범위: 기간단위별 허용 옵션

### 처리 규칙
- 기간범위 옵션은 기간단위에 따라 달라져야 한다.
- 기간단위 변경 시 기간 기반 동적 필터는 새 기준에 맞게 재계산되어야 한다.
- 기간단위 변경은 단순 표시 변경이 아니라, 결과 컬럼 축과 조회 범위를 바꾸는 상태 전이다.

### 출력/결과
- base period 집합
- 이후 기간 필터가 사용할 후보 축

## 9.3 측정단위
### 목적
- 비교할 집계 레벨을 정의한다.

### 입력
- source에서 제공되는 `dimension_type`

### 처리 규칙
- 측정단위 목록은 고정 상수보다 source의 실제 값에 우선해야 한다.
- 동일한 UX 명칭이라도 내부 조회 경로가 legacy serving인지 expanded/source query인지 다를 수 있다.
- 신규 `dimension_type`가 source에 생기면, UI와 provider 매핑 정책에 따라 노출 여부를 판단한다.

### 출력/결과
- 현재 결과의 entity grain
- 엔티티 필터 계산 기준
- 엔티티 드릴다운 가능 후보

## 9.4 동적 필터
### 목적
- 사용자가 시간과 대상 범위를 단계적으로 좁혀 탐색하도록 한다.

### 입력
- 현재 기간단위
- 현재 측정단위
- 현재 선택된 다른 필터들
- drilldown context

### 처리 규칙
- 기간 필터와 엔티티 필터는 동적으로 계산되어야 한다.
- 각 필터는 서로 연동되어야 한다.
- 하나의 필터가 바뀌면 다른 필터 후보도 현재 조건에 맞게 다시 계산되어야 한다.
- 기본 진입 시 각 필터는 전체 선택 상태를 갖는다.
- `전체 선택`은 제한 없음이다.
- `0개 선택`은 빈 결과 조건이다.
- 현재 옵션이 좁혀진 상태에서 남은 옵션을 전부 선택해도, UI는 실제 남은 값을 요약해 보여줄 수 있다.

### 출력/결과
- 현재 조건에서 선택 가능한 필터 후보
- heatmap/raw-data/드릴다운 조회에 사용될 실제 조건 집합

### 예외/주의사항
- `전체 선택`과 `0개 선택`을 혼동하면 안 된다.
- 동적 필터 로딩 중 stale 상태로 잘못 조회하지 않도록 auto-refresh 시점을 제어해야 한다.

## 9.5 조회 실행
### 목적
- 현재 검색 조건을 결과 데이터로 변환한다.

### 입력
- 지표, 기간단위, 기간범위, 측정단위, 필터 선택 상태
- drilldown context

### 처리 규칙
- 첫 조회 전에는 사용자가 명시적으로 조회를 시작한다.
- 결과가 이미 존재하는 상태에서는 조건 변경이 자동 재조회로 이어질 수 있다.
- 자동 재조회는 사용자 입력에 의해 예약되어야 하며, 단순 로딩 완료는 재조회 조건이 아니다.
- 재조회가 여러 번 경합할 경우, 마지막 사용자 의도를 기준으로 결과가 유지되어야 한다.
- drilldown 상태에서는 루트 조회로 되돌아가면 안 된다.

### 출력/결과
- 결과 테이블 데이터
- 기간 컬럼 축
- 엔티티 결과 집합

## 9.6 결과 테이블
### 목적
- 사용자가 선택한 조건의 결과를 비교 가능한 구조로 본다.

### 처리 규칙
- 결과는 기간 축과 엔티티/지표 축을 가진 표 구조로 제공한다.
- 기간 필터는 조회 조건뿐 아니라 결과 컬럼 축에도 반영되어야 한다.
- heatmap, 추이, 증감은 같은 결과 데이터 위에서 동작해야 한다.

### 출력/결과
- 행 단위 비교
- 기간별 값 비교
- 시각적 상태 표현

## 9.7 히트맵
### 목적
- 수치의 상대적 크기를 빠르게 비교한다.

### 처리 규칙
- 같은 지표 내부의 최솟값/최댓값 기준으로 색상을 계산한다.
- 미완료 기간은 heatmap 강조에서 제외할 수 있다.
- 사용자는 지표별 색상 on/off 또는 색상 변경이 가능해야 한다.

## 9.8 추이
### 목적
- 각 행의 시계열 흐름을 빠르게 파악한다.

### 처리 규칙
- 스파크라인은 현재 결과 컬럼 축 순서를 따른다.
- 추세선은 완료된 기간 값만 사용해 계산한다.
- 하락 추세는 음수 기울기 기준으로 시각적으로 구분한다.
- 미완료 기간은 스파크라인 점/추세선 계산에서 제외한다.

### 예외/주의사항
- 표시된 값은 있어도 미완료 기간이면 추이에 포함되면 안 된다.

## 9.9 증감
### 목적
- 이전 기간 대비 변화량을 본다.

### 처리 규칙
- 직전 완료 기간과 비교해 증감을 계산한다.
- 숫자형 지표는 `%`, 비율형 지표는 `%p` 기준으로 보여준다.
- 사용자는 지표별 증감 노출 여부를 제어할 수 있어야 한다.

## 9.10 정렬
### 목적
- 특정 기간 값 기준으로 결과 순서를 바꿔 비교를 쉽게 한다.

### 처리 규칙
- 정렬 기준은 기간 컬럼 단위다.
- 정렬 아이콘은 hover 없이도 보이는 것이 원칙이다.
- 기간 드릴다운 클릭 영역과 정렬 클릭 영역은 분리되어야 한다.

## 9.11 엔티티 드릴다운
### 목적
- 더 세부 단위로 내려가 원인을 찾는다.

### 입력
- 현재 엔티티
- 현재 측정단위
- 현재 필터/기간 조건

### 처리 규칙
- 드릴다운 후보는 단순 계층 관계가 아니라 실제 데이터가 존재하는 후보만 노출한다.
- 상위 parent context를 함께 반영해야 한다.
- 드릴다운 후에도 기존 필터 컨텍스트는 유지되어야 한다.

### 출력/결과
- 하위 측정단위 결과
- 엔티티 breadcrumb

## 9.12 기간 드릴다운
### 목적
- 특정 기간 하나를 더 세부 기간으로 분해해 본다.

### 입력
- 현재 기간 컬럼
- 현재 기간단위
- 클릭한 상위 기간 label

### 처리 규칙
- 기간 드릴다운은 상단 기간단위 변경과 다른 기능이다.
- 클릭한 기간의 시작/종료 범위 내부에 포함되는 하위 기간만 대상으로 해야 한다.
- 엔티티 드릴다운과 동시에 유지될 수 있어야 한다.
- 표시 라벨과 내부 key가 다르더라도 원본 period를 올바르게 식별해야 한다.

### 출력/결과
- 하위 기간단위 결과
- 기간 breadcrumb

## 9.13 미완료 기간 처리
### 목적
- 아직 집계가 끝나지 않은 값을 완료 수치처럼 오해하지 않도록 한다.

### 처리 규칙
- `오늘`이 포함된 기간부터는 미완료 기간으로 간주한다.
- 일 단위에서는 `오늘`
- 주/월/분기/연 단위에서는 현재 진행 중인 기간 전체
- 미완료 기간은 흐리게 표시한다.
- 미완료 기간은 추이와 추세선 계산에서 제외한다.
- AI 분석에서도 기본적으로 제외 대상이다.

## 9.14 템플릿
### 목적
- 자주 쓰는 검색 조건을 재사용한다.

### 처리 규칙
- 템플릿은 단일 값이 아니라 복합 검색 상태를 저장해야 한다.
- 지표, 기간, 측정단위, 필터 선택 상태를 함께 저장해야 한다.
- 구버전 구조가 있으면 호환 복원 전략이 필요하다.

## 9.15 데이터 내보내기
### 목적
- 조회 결과를 외부 분석이나 공유에 활용한다.

### 처리 규칙
- 현재 조회 조건 기준 결과를 내보내야 한다.
- 가능하면 결과 테이블 외 원본 데이터 시트도 함께 제공할 수 있다.

## 9.16 Kevin AI
### 목적
- 사용자의 수동 해석 시간을 줄인다.

### 처리 규칙
- AI는 현재 조회 결과를 기준으로 분석해야 한다.
- 미완료 기간은 제외하고 분석하는 것이 기본 정책이다.
- AI는 추이, 상위/하위 엔티티 비교, 이상치 설명을 지원해야 한다.

### API 공통 원칙
- 기존 응답 shape를 함부로 깨지 않는다.
- 프론트의 상태 모델과 어긋나는 응답 구조 변경은 금지한다.
- 캐시가 있더라도 드릴다운이나 최신성 요구가 높은 경로는 예외 처리할 수 있어야 한다.

## 12. 성능 및 캐시 정책
- 필터 옵션, 측정단위 목록, 기간 목록은 반복 호출이 많아 캐시를 허용한다.
- 결과 조회는 정확도가 우선이다.
- stale 캐시 때문에 현재 조건과 다른 결과가 보이면 안 된다.
- batch API를 활용해 필터 축별 순차 호출을 줄인다.

## 13. 검증 기준
### 13.1 기본 검증
- `npm run build`
- `git diff --check`

### 13.2 기능 검증
- 지표 선택 후 조회가 정상 동작해야 한다.
- 기간단위/기간범위 변경 시 결과 축이 바뀌어야 한다.
- 동적 필터는 cascade 방식으로 재계산되어야 한다.
- 결과 표시 후 필터 변경은 자동 재조회가 정상 동작해야 한다.
- 엔티티 드릴다운과 기간 드릴다운은 컨텍스트를 유지해야 한다.
- 미완료 기간은 흐리게 표시되고 추이에 포함되지 않아야 한다.

### 13.3 데이터 검증
- source와 serving 간 누락이 없어야 한다.
- 대표 지표 기준 최신 완료 기간 노출이 정상이어야 한다.
